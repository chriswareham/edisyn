Below is a little spreadsheet of sorts to be used as a record for
reverse engineering the locations of parameters in a Hydrasynth
patch sysex dump.  This patch format is entirely reverse engineered
(ASM has not been forthcoming) and may have errors, though it appears
to be fairly complete for versions 1.5.5 and 2.0.0.

The patch sysex data is exposed by decoding a series of patch dump
sysex messages.  This procedure is described here:

https://github.com/eclab/edisyn/blob/master/edisyn/synth/asmhydrasynth/sysex/SysexEncoding.txt

This patch format was used to build Edisyn's Hydrasynth patch editor.


OVERVIEW AND EXPLANATION OF TABLE

It appears that the first eight bytes are NOT part of the patch, but
rather are metadata.

Most parameters are set up to consume two bytes.  When the parameter
is less than 8 bits, it uses the even byte position.  However if such a
parameter represents a negative value, then it is sign-extended into the
second byte (that is, it should be treated as a two-byte negative number).
When the parameter requires more than 8 bits, the even byte position 
defines the 8-bit LSB of the parameter and the (following) odd byte 
position defines the 8-bit MSB of the parameter.

Strings are one byte per character, and may take up and odd amount of space 
(see the apparently 17-byte Macro Name space for example).  A few parameters 
seem to be crammed into single byte with no additional byte, such as 
Category (byte 8), whose "MSB" space is used by the Patch Name Start, and also
the various Macro Panel Button Values.  Color may be a similar parameter, it's 
not quite clear.  Also see byte 470, which uses bitpacking to stuff both Warm 
Mode and Snap into the same byte. There's also a hack employed for Vibrato 
Amount to enable higher resolution for 2.0.0: byte 46 has the integer value 
as usual, but byte 2462 now holds the decimal value.

For the ranges of parameters, see the NRPN spreadsheet found here:
https://github.com/eclab/edisyn/tree/master/edisyn/synth/asmhydrasynth/info

It is not recommended to use ASM's provided NRPN file, 
"Hydrasynth_KB_DR_MIDI_Spec_1.5.0.pdf", which is both incomplete and which
entirely lacks display information.

There are regions whose purpose or value is unknown an are marked either
as UNKNOWN or as UNKNOWN START ... UNKNOWN END.

The documentation in this table is sparse: there are after all over 1000
parameters in the Hydraynth.  If data repeats, then only the first data
elements are indicated, plus a few boundary examples so you can get a
handle on how the repeating data works.

The Hydrasynth appears to be very unusual in that in many cases two versions 
of a parameter fill the current working memory, but only one is written to 
the patch.  Specifically, the Hydrasynth has BPM Sync ON and BPM Sync Off
versions for a number of parameters.  In current working memory these are
two SEPARATE parameters with their own NRPN addresses.  But when writing the
patch to storage or dumping it, the two parameters are collapsed to a
single parameter depending on whether BPM Sync is on or off.  It's kind of
like Schrodinger's Cat.
